## 検討と意思決定 (Decisions & Rationale)

### 1. interaction/SKILL.md は最初から voice 拡張込みで書く

- **判断:** voice/listen/say を**最初から interaction/SKILL.md に統合**。Step 1.5 として後付けしない
- **理由:** 後から音声を追加すると plan_response_tool の `voice` フィールドの扱いが二重定義になりがち。voice 入力時の `voice.speak` 上書きルールが SKILL.md 内で散らばらないようにする
- **代替案:** voice 拡張を別 SKILL に切る → 不採用（フェーズ単位ではなくタスク単位という前セッションの方針に反する）

### 2. paths は YAML 配列ではなくカンマ区切り

- **判断:** `.claude/rules/*.md` の frontmatter `paths:` フィールドは**カンマ区切り**で書く
- **理由:** 参照記事（[Qiita](https://qiita.com/nogataka/items/d6c83ea50b82e1c2602c)）のバグ指摘。YAML 配列形式だとパースが正しく動かない
- **適用例:**
  - 正: `paths: src/api/**/*.ts,src/routes/**/*.ts`
  - 誤: YAML 配列形式

### 3. CLAUDE.md は 45 行で妥協

- **判断:** 目安は 30-35 行だが、**45 行**（三層モデル表とディレクトリマップ込み）で妥結
- **理由:** 三層モデル表は設計の中核なので削れない。ディレクトリマップも MCP 構成把握に必須。45 行は許容範囲
- **代替案:** 三層モデルを別ファイル（`.claude/rules/three-layer.md`）に切る → 不採用（rules はパス条件で注入されるが、三層モデルは**全コンテキストで読まれるべき設計原則**なので CLAUDE.md に残す）

### 4. rules ファイルは MCP サーバー単位 + 横断規約 2 件

- **判断:** 9 ファイル構成（MCP ごと 7 つ + python-mcp-dev / mcp-server-design 2 つ）
- **理由:** ツール一覧は MCP 単位で読みたい（編集中の MCP に応じて自動注入される）。一方 uv/pytest/ruff の開発フローと .env / mcpBehavior.toml / jurigged の規約は MCP 横断で共通
- **代替案:** 1 ファイルにまとめる → 不採用（CLAUDE.md 肥大化と同じ問題が rules 側で再発する）

### 5. Step 5 メタ層は「継続改善 + 観測ベース着手」方針

- **判断:** Step 5 は今すぐ実装せず、運用データを集めてから判断するタスクとして HANDOVER に登録
- **理由:** メタ層は「自己認識補助 / 行動は強制しない」原則。観測なしに自動化を入れると過剰防御になる
- **観測トリガー（Plan エージェントの提案）:**
  - followup pending: `agent_experiences.count < plan_calls_with_followup × 0.5` が 1 週間続く
  - turns_since_last_experience: record なしの outgoing が 5 連発するセッションが週 2 回以上
  - interpretation_shift drift: shift 後 7 日以内に同種の user_correction が再発
- **代替案:** 即実装 → 不採用（auto-social.sh の轍を踏むリスク。先に観測スクリプトを書く）
