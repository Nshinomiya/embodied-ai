## 検討と意思決定 (Decisions & Rationale)

- **判断:** screen-read 実行中の許可プロンプトを「画面スクロール時のユーザー確認」に集約し、その他のシェル/ファイル操作は permission で先回り許可する
  - **理由:** screen-read は「PgDn → 撮影 → OCR → 疑義提示」のループ。撮影前の物理操作（PgDn / 「次へ／END」応答）はユーザー必須だが、ファイル コピー・テキスト抽出・補正テキスト書き戻し等は機械的に流れるべき。許可プロンプトが間に挟まると体験が崩れる
  - **代替案:** 全部許可（広すぎ）／ 撮影前以外も都度許可（体験が悪い）

- **判断:** `.text` 取り出しは `jq -r '.text' <file>` に統一し、permission は `Bash(jq -r *)` で広めに許可
  - **理由:** 可読性（一目で意味が分かる）、helper 改修不要、`uv run --project ...` の起動オーバーヘッドを避けたい。jq は Claude Code の statusline 表示でも使われており、インストール前提として扱える
  - **代替案 A**（helper に `get-text` サブコマンド追加）: 処理が helper.py に集約される設計的純度はあるが、コード追加と起動コストの増加が見合わない
  - **代替案 B**（`Bash(python3 -c *)` を許可）: 過剰すぎて screen-read 以外でも全 python ワンライナーが通る

- **判断:** `Bash(jq -r *)` のスコープは絞らず広めに残す（`Bash(jq -r '.text' /tmp/screen-read-*)` までは絞らない）
  - **理由:** jq は read-only（書き込み副作用なし）、可読性・将来の `.boundaries` 等を確認したい場面まで含めて運用したい
  - **代替案:** ファイルパス限定（screen-read 専用に縛る）— 安全側だが将来 merge 結果から `.boundaries[].uncertain` を見たい時に毎回プロンプトが出る

- **判断:** `rm -rf /tmp/screen-read-*` パターンを許可
  - **理由:** SKILL.md L34 で Claude が `mkdir -p /tmp/screen-read-${SESSION_ID}` を作る規約。同パターン配下しか削除しないので事故の範囲が screen-read 自分のセッション dir に限定される。`/tmp/wifi-cam-mcp/`（wifi-cam-mcp サーバー管理）は別パスで影響しない

- **判断:** 撮影前のユーザー応答待ち（「次へ／END」確認）は permission ではなく SKILL の対話設計で担保
  - **理由:** `mcp__wifi-cam__see` を permission allow しても、Claude が see を呼ぶ前にプロンプトを出す挙動は変わらない。これは Claude 側の判断順序で、SKILL.md L146 に明記済み。permission で締めるべきは「ユーザーの心の確認が必要な操作」だけ

## ハマった点・失敗したアプローチ (Friction & Anti-patterns)

- **問題:** Gemini Pro が表セルの「画面端 word-wrap」を「セル A1 の `<br>` 続き」と誤読する（page-4 で全行発生）
  - **試したこと:** Flash でクロスチェック → Flash も同様の誤読。「ユーザー」「回答」「検索語」のように理由列セル先頭の数文字だけが「理由列」に、続きの長文が「1 列目セルの `<br>` 続き」に分断される
  - **結果:** OCR モデル単体では構造復元できない。ユーザーが画面の物理レンダリング（折返し）を教えてくれて初めて論理セル化が可能になった。**次回**: 表が出る画面では撮影前にユーザーに「表セルが折り返されているか」確認するステップを SKILL に追加できると安全

- **問題:** Pro が ` ``` ` コードフェンスを `---` と読む癖 → merge で前ページ末尾の正規 `---` と衝突してコードブロックの開始フェンスと「【回答ルール】」見出しが消失（page-7 → page-8 境界）
  - **試したこと:** save-page の段階で page-8 側を ` ``` ` に修正したが、merge.py の overlap 検出が前ページ末尾の `---` を「page-8 先頭の `---`」と同一視
  - **結果:** 自己確認段階で Edit で復元。**次回**: コードブロックを含むページは save-page 前に「前ページ末尾の `---` の有無」をチェックし、必要なら前ページ側で末尾の余分な `---` を削るほうが安全

- **問題:** merge の overlap 誤判定（boundary 8: page-9 → page-10、`k=15 (drop next first 0+15 lines)`）でチェックリスト 2 項目が削除（5 項目目「エージェントの『指示』にテーブル構造への言及禁止…」と 6 項目目「シノニム・用語集を設定している」）
  - **試したこと:** merge は前ページ末尾と次ページ先頭の overlap を貪欲にマッチさせるが、`## 10. チェックリスト + 4 項目` が両ページに存在したため、page-10 固有の 5-8 項目目まで巻き込んで削除
  - **結果:** 自己確認時に発見して Edit で復元。**次回**: merge 後の自己確認では「チェックリスト等の項目数が page-N と merge 結果で一致するか」を必ず照合する。`screen_read/merge.py` の `MIN_SCORE=91.0` / 探索範囲を `k` 大きい方向に偏らせない閾値見直しが本質的解決

- **問題:** 章境界（`### 見出し` の前後）で空行が消える（5 箇所）
  - **試したこと:** Pro の OCR 出力は空行を保持しているが、merge の overlap 削除で前ページ末尾の空行と次ページ先頭の見出しの間が詰まる
  - **結果:** 自己確認時の Edit で復元（行 39/61/118/142/189）。**次回**: 自己確認チェックリストに「`### ` の前後に空行があるか」を明示

- **問題:** OpenRouter から `content: None` が返って `_strip_outer_fence(None)` が AttributeError（page-3 で発生）
  - **試したこと:** SKILL.md L98 の既知事象。同じ画像で 1 回手動リトライ
  - **結果:** リトライで通った。helper の F-16 リトライ（HTTP 429/5xx）はカバーしているが、`content: None` は別経路。**次回**: helper 側で `content is None` を 1 回だけリトライする実装を入れると体験が安定する

- **問題:** OCR を `run_in_background=true` で投げると hang して 17 分待たされた（page-6 でも background→completed 通知が来たが、page-10 では完了しないまま中断）
  - **試したこと:** OCR が時間かかると判定されてバックグラウンド化
  - **結果:** page-10 はフォアグラウンド（`timeout: 300000`）で実行したら数秒で完了。**次回**: OCR は基本フォアグラウンド + timeout 300000 ms で固定する。バックグラウンド化は notification が来ないと検知遅延が大きい

- **問題:** Markdown プレビューの下に Copilot チャット応答（「全体の議論を網羅的に整理しました…」）が映り込み、page-10 の OCR が本文として拾った
  - **試したこと:** F-15 のシステム指示（オーバーレイ無視）はあるが、画面下部の連続テキストブロックとして区別がつかなかった模様
  - **結果:** save-page 前に Python で末尾分割して切り捨て。**次回**: 撮影前に「画面下部にチャット応答や別 UI が映っていないか」を確認するステップが SKILL にあると安全
