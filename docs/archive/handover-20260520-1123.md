## 検討と意思決定 (Decisions & Rationale)

### 1. mirrored networking の深追いをやめて案 B に切替

- **判断:** WSL2 mirrored mode のトラブルシューティングを打ち切り、VOICEVOX 側を `0.0.0.0` バインドに変更する**案 B** を次の最短ルートとする
- **理由:** 環境調査の結果、`.wslconfig` も WSL バージョン（2.4.12.0）も Windows ビルド（26200.8246）も mirrored 対応のはずなのに NAT モードのまま。Windows 11 24H2 build 26200 系で mirrored が効かない既知問題の可能性。Hyper-V vSwitch / Windows Defender Firewall まで踏み込む価値より、VOICEVOX 1 設定変更で済ませる方が ROI が高い
- **代替案:**
  - 案 A（mirrored 修復）: Windows 再起動 / `wsl --update` / Hyper-V Firewall プロファイル調整。Windows 環境深掘り必要、不確実
  - 案 C（Windows portproxy）: 永続化が面倒で再起動で消える可能性。お勧めしないと判断
- **副作用:** 案 B は VOICEVOX が外部からも接続可能になる。LAN 上の他端末からも 50021 に届くようになるが、VOICEVOX は認証なし API なので **LAN 内に信頼できない端末がある環境では非推奨**。家庭 LAN 内なら問題なし

### 2. 案 A 切替を試したが効果なし（次セッションでは試さない）

- **判断:** ユーザーが `.wslconfig` に `networkingMode=mirrored` を追加 → `wsl --shutdown` → 再起動を行ったが無効。次セッションでは試さない
- **理由:** 上記環境の組み合わせで設定が反映されない事象は再現性がある（uptime=1min での確認でも NAT のまま）
- **再挑戦するなら:** Windows 自体の再起動 + `wsl --update` + `Get-NetFirewallHyperVProfile` 確認が必要。それでも mirrored が効かなければ別の構造的問題

## ハマった点・失敗したアプローチ (Friction & Anti-patterns)

### F-1: `.wslconfig` 正、WSL バージョン正、なのに mirrored が反映されない

- **問題:** `localhost:50021` への接続がタイムアウト、`192.168.10.1:50021`（Windows host IP）も connection refused
- **試したこと:**
  1. `.wslconfig` 中身確認 → `[wsl2]\nnetworkingMode=mirrored` で正、ASCII CRLF、48 bytes、BOM なし
  2. `%USERPROFILE%` の実値確認 → `C:\Users\SlmbrCat`、`.wslconfig` の場所と一致
  3. WSL バージョン確認 → 2.4.12.0（Store 版 + 機能版両方インストール済み、両方 mirrored 対応版）
  4. カーネル → 5.15.167.4（mirrored 対応）
  5. `wsl --shutdown` → 再起動（uptime=1min で確認）
- **結果:** `ip addr` に Windows 側 NIC が見えない（`eth0` のみ、NAT モードの典型）。`/etc/resolv.conf` の `nameserver 10.255.255.254` は WSL2 NAT 既定 DNS。**設定が反映されていないことが確定**
- **学び:** `.wslconfig` の形式が正しく WSL バージョンが新しくても mirrored が効かないケースが存在する。`/etc/resolv.conf` の nameserver と `ip addr` を見るのが最速の判定

### F-2: `wsl --status` の表示形式が古いまま

- **問題:** `wsl --status` の出力に `Default Networking Mode` 行が出てこない（古い形式）
- **試したこと:** Store 版（`C:\Program Files\WSL\wsl.exe`）と機能版（`C:\Windows\System32\wsl.exe`）両方で `--status` 実行
- **結果:** どちらも古い形式の表示（「WSL1 はサポートされていません」のみ）
- **学び:** Store 版のバイナリ（3.9MB）と機能版（258KB）は別物だが、`--status` の出力では networking mode を確認できない。確認には `ip addr` と `/etc/resolv.conf` を見るのが確実

### F-3: 最初の curl が遅延 timeout で原因切り分けが遅れた

- **問題:** 最初の `curl localhost:50021` で IPv6 (`::1`) が即時 connection refused、IPv4 (`127.0.0.1`) で 3 秒タイムアウト
- **学び:** `curl -v` で IPv6/IPv4 両方の挙動が見える。次回類似事象では最初から `-v` を付ける
