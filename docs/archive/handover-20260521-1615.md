## 前提と目的 (Context & Intent)

embodied-ai プロジェクト「テキスト＋音声インタラクティブ会話」マイルストーンに向けた Phase 1（VOICEVOX 接続）の続行。前セッションまでに以下を確認済み：

- WSL2 mirrored mode は Windows 11 24H2 build 26200 + ESET 環境で反映されない（[[wsl2-mirrored-mode-が反映されないケースの判定と回避]]）
- Windows 側 VOICEVOX を `0.0.0.0` バインドにしても WSL から `localhost:50021` に届かない（NAT ルーティング自体が破損）
- 解決方針：WSL 内に Docker をインストールし、VOICEVOX を WSL 内コンテナで起動して localhost で完結させる

今回はその「WSL 内 Docker で VOICEVOX を立ち上げる」運用 step を実機で完遂することと、tts-mcp が Claude Code から見えるようにする問題の解決が主目的。コード変更ゼロ、運用・調査のみ。

## 検討と意思決定 (Decisions & Rationale)

- **判断:** VOICEVOX を WSL 内 Docker で動かす方針に確定
  - **理由:** WSL→Windows host の NAT ルーティングが Windows 11 24H2 build 26200 + ESET 環境で構造的に壊れており、Windows 側 `0.0.0.0` バインドでも疎通不可。WSL 内に閉じれば「同一 Linux 名前空間 + localhost」で確実に繋がる
  - **代替案:** (1) mirrored mode を深追い → 解決保証なし、時間 ROI が悪い。(2) Windows portproxy → 再起動で消える、管理者権限必要、復旧手順複雑。(3) Docker Desktop 経由 → ライセンス問題回避と Docker Engine native の方が単純

- **判断:** GPU 版でなく CPU 版イメージを採用
  - **理由:** WSL の GPU passthrough（NVIDIA Driver 576.02、CUDA 12.9）は動作確認済みで GPU 版も viable だが、まず素疎通を最短で取りに行く。CPU 版でも対話用途では合成速度実用範囲、イメージサイズ約 5GB（GPU 版 ~10GB の半分）
  - **代替案:** GPU 版で最初から低レイテンシ確保 → セットアップ手順が増え、Phase 1 完遂が遅れる。必要になったら後から `docker rm voicevox && docker run ... nvidia-ubuntu20.04-latest --gpus all` で差し替え可能

- **判断:** tts-mcp が `/mcp` に出ない問題は `claude mcp reset-project-choices` で対処
  - **理由:** `claude mcp list` で他の MCP は表示されるが tts だけ非表示、`~/.claude.json` の project entry の `enabledMcpjsonServers: []`・`disabledMcpjsonServers: []`・`enableAllProjectMcpServers: None` から、後から追加された tts が過去の trust 承認の対象外であることが判明。Claude Code は新規 .mcp.json エントリに対して個別承認を要求する設計だが、その承認プロンプトが表示されていなかった
  - **代替案:** `~/.claude.json` を手動編集 → スキーマ確定していない、CLI の正規手段がある。`claude mcp get tts` は trust dialog をスキップして spawn する診断用で、通常セッションの有効化には繋がらない

## ハマった点・失敗したアプローチ (Friction & Anti-patterns)

- **問題:** 初回 `docker run` で `failed to bind host port 0.0.0.0:50021/tcp: address already in use`
  - **試したこと:** WSL 内で `ss -ltnp 'sport = :50021'` 実行 → 何も listen していない
  - **結果:** Windows host 側で動いていた VOICEVOX が、mirrored mode の部分的挙動 / Hyper-V vSwitch を経由して WSL 側の 0.0.0.0:50021 バインドと衝突していた。Windows GUI 版 VOICEVOX を完全終了させて解決
  - **教訓:** Windows 側で同一サービスが残っていると、WSL 内 Docker のポートバインドが見えない経路でブロックされる。Docker 起動前に Windows host 側のサービス停止必須

- **問題:** Docker 直起動が `permission denied while trying to connect to the docker API at unix:///var/run/docker.sock`
  - **試したこと:** `usermod -aG docker $USER` 後に Claude Code セッション内 bash で `docker run` 実行
  - **結果:** Claude Code が起動した bash プロセスのプロセスツリーは usermod 前のグループのままなので、グループ反映されていない。`sg docker -c "docker ..."` で都度サブシェルにグループを切替えて回避
  - **教訓:** Claude Code セッション内から Docker を叩くときは `sg docker -c "..."`。完全な反映には Claude Code 再起動か WSL 再起動

- **問題:** tts-mcp が `/mcp` 一覧に表示されず Claude Code 再起動でも改善しない
  - **試したこと:** (1) `.mcp.json` の構文確認（OK）、(2) `uv run --directory tts-mcp tts-mcp` 手動起動（env 渡せば stay-up、exit 0）、(3) ElevenLabs プレースホルダ key で起動失敗していないか確認（config.py の遅延初期化なので無関係）、(4) Claude Code 再起動 2 回
  - **結果:** 全部正常だったが Claude Code 内では tts 未認識。原因は `~/.claude.json` の project entry に tts が未登録だったこと（後から追加されたエントリの個別承認が抜けていた）

- **問題:** `ELEVENLABS_API_KEY="your-api-key-here"` プレースホルダが残っている
  - **試したこと:** 起動失敗の犯人と疑った
  - **結果:** 関係なし。tts-mcp は `_get_client()` で遅延初期化するので、fake key でも起動自体は通る。実際の発話時に 401 になるが、それまで触らなければ問題ない。VOICEVOX-first 運用なら放置可
