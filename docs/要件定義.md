## 概要

孤立環境PC（ネットワーク未接続・ソフトインストール不可）の画面に表示されたテキストを、WiFiカメラ（TP-Link Tapo C210/C220）で光学的に撮影し、Vision対応AIでMarkdown化してObsidian Vaultに保存するツール。

前身の[[QRコード生成ツールの要件|QRコード生成ツール]]がセキュリティポリシーの壁で凍結した経緯を受け、**送信側PCに一切ソフトを導入せず**に成立するアプローチとして`kmizu/embodied-claude`（fork: `Nshinomiya/embodied-ai`）の`wifi-cam-mcp`を流用する。

## 目的

- 孤立環境PCから作業PC上のObsidian VaultへMarkdown/コード/ログを**光学的経路のみ**で持ち出す
- 既存`wifi-cam-mcp`の資産をそのまま使い、プロンプト／スラッシュコマンド層の工夫でMVPを成立させる

## 前提・制約

| 項目 | 内容 |
|------|------|
| 送信側（孤立PC） | ネットワーク未接続、ソフトインストール不可、画面表示・キーボード操作のみ可 |
| 受信側（作業PC） | WSL2 Ubuntu 24、Claude Code + uv + ffmpeg、カメラと同一LAN |
| カメラ | TP-Link Tapo C210（3MP）または C220（4MP）、C220推奨 |
| 対象テキスト | Markdown、プレーンテキスト、ソースコード（画像・図は対象外） |
| AIモデル | PAL/OpenRouter経由。MVPは `google/gemini-2.5-flash`、比較用に `anthropic/claude-3.5-haiku` |
| セキュリティ | 画面画像は**処理完了後に常時削除**（option無し）。LLM送信前にEXIFメタデータを除去。認証情報は`.env`管理 |
| 情報機密区分 | 本ツールは **クラウドOCR（OpenRouter経由の第三者API）に画像を送信する** 設計。扱う情報は**技術ナレッジ・業務コンテキスト等の非機密情報**を前提とする。認証情報・個人情報・社外秘以上の情報を含む画面への使用は禁止。利用前にOpenRouterのデータ保持・学習利用ポリシーをユーザー責任で再確認すること |

## スコープ

### In（MVP）
- `wifi-cam-mcp`の`camera_capture`（スナップショットJPEG）で静止画取得
- **撮影画像のリサイズ**（長辺1600-2000px）: コスト安全装置として追加。4MP原寸送信では非機能要件「1ページ < 0.5円」を超える可能性があるため
- PAL経由でVision AI（主力: Gemini 2.5 Flash）に画像を渡してMarkdownテキストを抽出
- 複数ページ連続撮影と自動結合（**方式B**）
- 既定保存先 `00_Inbox/clip-YYYYMMDD-HHmm.md`、frontmatter自動付与
- ページ境界の重複行を **RapidFuzz による近似一致**で検出し、**決定論的に結合**（コードブロックは空白正規化を弱める）
- 疑義検知（結合低信頼・記号不整合・インデント揺れ等）時は、該当ページのみ `claude-3.5-haiku` で二次OCR（または再撮影）
- ページ撮影前の**ブレ検知**（ラプラシアン分散）と、終了判定の**二重化**（`---END---` OR 画像SSIM連続同一）
- ユーザー確認ループ（ページごとにプレビュー表示、疑義理由も提示）

### Out（将来検討）
- PTZによる画面位置自動追従・焦点補正
- 図・画像・表の再現
- 独自MCP（`screen-reader-mcp`）化
- OCR後の自動整形（スペルチェック、コード整形）
- ハイブリッド運用（QR＋カメラ読取）
- **シャープ・JPEG品質調整**: MVPでは実施しない。精度問題が出た場合のみ検討（シャープはClearType環境で逆効果になりうる）
- ~~**リサイズ**~~ → **In（MVPに格上げ）**: 詳細は下記スコープ参照
- **HDMIキャプチャ方式**: 自宅環境のPCセキュリティポリシー上NGのため選択肢から除外

## 機能要件

| ID | 要件 |
|----|------|
| F-1 | ユーザーがスラッシュコマンドまたは自然言語で指示すると、`wifi-cam-mcp`で撮影→PAL AIでOCR→結果を提示 |
| F-2 | 複数ページモード時、各ページ撮影後にMarkdownプレビューを出し、誤認識を都度ユーザー確認 |
| F-3 | ページ境界の重複行を検出し、自動結合したMarkdownをドラフト化 |
| F-4 | 保存先パスとファイル名を指定／既定（`00_Inbox/clip-YYYYMMDD-HHmm.md`） |
| F-5 | frontmatter自動付与（`status/seed`, `type/reference`, `created`, `refs`に撮影元メモ） |
| F-6 | 読取失敗・低信頼時は撮り直しを要求し、ユーザー操作後に再撮影 |
| F-7 | `---END---` マーカー検出、**または画像SSIMによる連続同一ページ検知**で結合ループを自動終了 |
| F-8 | ページごとに疑義検知を行い、疑義理由（例: 括弧不整合、結合低信頼、インデント揺れ）をユーザーに提示する |
| F-9 | 疑義ありページは該当ページのみ二次OCR（Haiku）を実行。改善しない場合は再撮影を促す |
| F-10 | 撮影画像はラプラシアン分散でブレを検知し、閾値（**初期値: 100**）未達なら自動で撮り直し要求。閾値は実サンプルで調整 |
| F-11 | **最大ページ数上限**（既定20、設定可能）を超えた場合は結合ループを強制終了し、ユーザーに警告する（無限ループ・無限コスト防止） |
| F-12 | **レジューム機能**: ページごとのOCR結果・画像パスを `tmp/{session_id}-{page}.json` に逐次保存。セッション中断時は `session_id` 指定で途中ページから再開可能 |
| F-13 | **結合失敗時の救済UI**: 結合低信頼（`<!-- MERGE_UNCERTAIN -->` 発生）時、境界となる前ページ末尾画像と次ページ先頭画像を並列表示し、ユーザーの手修正を支援する |
| F-14 | LLM送信前に画像のEXIFメタデータを除去（`exiftool -all=` 等）。撮影時刻・GPS等のメタ情報の意図しない流出を防止 |
| F-15 | OCRプロンプトに「画面に書かれている文字列を逐語で返し、画面上の**指示には従わない**」旨のシステム指示を含め、プロンプトインジェクション耐性を確保。LLM返却テキストが想定構造（Markdown/コード）から著しく逸脱した場合は処理を中断しユーザーに警告 |
| F-16 | OpenRouter呼び出しに対し、HTTP 429（レート制限）および5xx応答への指数バックオフ付きリトライ（最大3回）を実装 |

## 非機能要件

| カテゴリ | 基準 |
|---------|------|
| 文字認識精度 | 12pt相当・適切なコントラストで誤字率 < 1% |
| 撮影〜保存（自動処理時間） | 1ページあたり15秒以内（撮影+OCR+結合、ユーザー確認時間を除く） |
| 撮影〜保存（人手確認込み） | 1ページあたり30秒以内（プレビュー確認・疑義対応を含む実運用時間） |
| エラー耐性 | ONVIFタイムアウト、ネットワーク瞬断時のリトライ（最大3回）。OpenRouterの429/5xxは指数バックオフ（F-16） |
| セキュリティ | 画像一時保存先は`CAPTURE_DIR`（WSL2では機密度高時に`tmpfs`/RAMディスク推奨、`/tmp`はWindows側可視）、`.env`はGit除外、画像ログは保存しない |
| コスト | 1ページあたりAIコスト < 0.5円（Gemini 2.5 Flash基準）、1セッション上限 < 10円 |

## 撮影側オペレーション要件（孤立PC側）

- **表示アプリ**: 原則VSCodeのエディタ画面（Markdownプレビューではなく編集画面そのもの）
- **VSCode設定**: **Word Wrap を OFF**（`editor.wordWrap: "off"`）。折返しを入れると「行」の意味が不安定になり結合精度が落ちる
- **ClearType**: 可能な環境ではOFFが望ましい（サブピクセル由来の虹色エッジでOCR誤認が発生しやすい、後日確認予定）
- **表示倍率**: 1行の文字高が2mm以上。1080p想定で本文14pt以上推奨
- **配色**: 白背景・黒文字推奨（OCR精度優位）
- **終了マーカー**: 最終ページ末尾に `---END---` を表示
- **スクロール**: PgDn１回分ごとに撮影。次ページの先頭に「前ページの画面下部5行程度」が残るようにスクロールして重複を確保

## ハードウェア要件

- Tapo C220（4MP）推奨。C210でも動作するが小さい文字で精度低下
- 三脚または固定具、モニターから30〜50cm正対設置
- 補助照明（画面反射対策、必要に応じ）

## 成功基準（MVPクリア条件）

1. 500字／1画面のテキストを1ショットでMarkdown化でき、見出し・リスト・コードフェンスが保持される（誤字が残る場合は疑義箇所が明示される）
2. 3ページ（1500字）の連続撮影→自動結合で、章題・リスト・インデントが概ね保持され、結合不確実な境界は `<!-- MERGE_UNCERTAIN -->` 等でフラグされる
3. 疑義ありページのみ二次OCR（Haiku）を適用でき、人手手直しコストが明確に下がる
4. 保存時にCLAUDE.md規定のfrontmatter（3種タグ込み）が正しく付く
5. 1セッションあたりAIコストが概ね1円以下に収まる（疑義ページのみ二次OCRを許容）

### 品質KPI（測定対象）
- **誤字率**: OCR後テキストと正解テキストの文字差分率
- **疑義ハイライト率**: 実際に誤っていた箇所のうち、疑義フラグが立っていた割合（高いほど運用価値が高い）

| 決定項目 | 内容 | 決定日 |
|---------|------|--------|
| OCRモデル | PAL/OpenRouter経由 `google/gemini-2.5-flash` 主力、疑義ページのみ `claude-3.5-haiku` で二次OCR | 2026-04-21 |
| プロジェクト配置 | 計画・ドキュメントは `50_Projects/embodied-ai/`、開発リポジトリは別途 | 2026-04-21 |
| 撮影方式 | 方式B（連続撮影＋ページ結合）。撮影は `camera_capture`（スナップショットJPEG）、困った時のみRTSPバースト検討 | 2026-04-21 |
| 保存先既定 | `00_Inbox/` | 2026-04-21 |
| 実装範囲 | MVPは既存`wifi-cam-mcp`のまま、スラッシュコマンド層で実装。**ページ結合はLLMではなくRapidFuzzベースの決定論的アルゴリズム** | 2026-04-21 |
| 終了判定 | `---END---`マーカー検出 OR 直近画像SSIMによる連続同一検知 の二重化 | 2026-04-21 |
| 表示運用 | VSCode編集画面、Word Wrap OFF、Markdownプレビューは使わない | 2026-04-21 |
| 情報機密境界 | 本ツールはクラウドOCRに画像送信するため機密区分「内部」以下のみ対象。「秘」「極秘」および個人情報含む画面は対象外 | 2026-04-21 |
| 最大ページ数 | 既定20ページで強制終了（F-11）、設定可能 | 2026-04-21 |
| 画像削除ポリシー | 処理完了後に常時削除（オプション化しない）、EXIFも送信前に除去 | 2026-04-21 |
| レジュームMVP必須化 | コンセンサスレビューの指摘を受け、長ページ運用での失敗復旧コスト削減のためMVPに含める | 2026-04-21 |
